01 Data Security · Real-Time Enforcement

AtherDLP Production-style DLP pipeline

Intercepts outbound HTTP traffic, classifies sensitive content, and enforces policy decisions (allow / alert / block) in near real time.

WHY THIS EXISTS: Security teams need near real-time control over outbound HTTP data flows with audit-grade telemetry and explainable allow/alert/block decisions.

  • Architecture: Built an outbound HTTP interception SDK that injects trace/span IDs, classifies destinations via O(1) registry lookup, and emits structured events for centralized enforcement decisions.
  • Data path: Normalized method/URL/headers/body (incl. multipart extraction) into a searchable corpus; redacted Authorization/Cookies to keep security logs safe and audit-ready.
  • Core logic: Implemented an AND/OR rule-graph policy engine over sensitive information types (regex + keyword matchers) with min_count thresholds and matched-rule evidence for explainability.
  • Latency vs side effects: Kept the decision path fast by decoupling alerts/actions via policy.triggered events + async workers; fail-open on telemetry to avoid blocking ingestion.
  • Reliability: Added Celery workers with 3 retries + 5s backoff, correlation IDs, and queued → processing → completed/failed status transitions for resilience under transient failures.

02 Machine Learning · Reproducible Inference

ChurnIt End-to-end churn prediction pipeline

Owned the full pipeline from raw data → features → training → reproducible inference, with a heavily optimized scoring path.

WHY THIS EXISTS: Product teams need reproducible churn scoring at scale with consistent train/serve transformations and debuggable predictions.

  • System design: Built an end-to-end pipeline (raw → features → training → inference) with deterministic preprocessing + feature generation to eliminate train/serve skew.
  • Performance: Rewrote scoring to be batch-first and vectorized (precomputed feature mappings, minimized redundant transforms, removed per-row Python overhead), improving throughput from ~100 predictions/5s → ~10,000 predictions/~5s (~100×).
  • Reliability tradeoff: Persisted versioned artifacts (model + preprocessing/feature steps + schema/config) loaded as a single immutable bundle for reproducible training, consistent online scoring, and safe rollbacks.
  • Explainability: Implemented SHAP global + per-prediction attributions tied to engineered features for debugging, validation, and trustworthy decisions.
  • Auditability: Logged data snapshots/versions, feature sets, metrics, and artifact checksums for auditable experiments and rollbacks without retraining.

03 C++ Systems · Storage & Search

Cognex Embedded KV store + BM25 search

Embedded C++ key-value store with durable persistence and a BM25-ranked inverted index powering full-text retrieval.

WHY THIS EXISTS: Infra/embedded workloads need durable KV + ranked retrieval with predictable latency and crash recovery under real constraints.

  • Storage core: Implemented a log-structured KV store with an in-memory hash index (Key → {value_log_offset, value_size}) over an append-only value log with CRC32 checksums; delivered ~503K GET ops/sec.
  • Durability: Added length-prefixed, checksummed WAL + atomic snapshots (temp file + rename + directory fsync); tuned durability with batched fsync every 1K WAL writes / 5K value-log appends.
  • Indexing: Built an inverted index (term → postings[]) storing {docId, tf, positions} with per-document aggregation; reached ~17.8K docs/sec indexing at 50K docs (RSS ~33 MB).
  • Query execution: Implemented BM25 ranking (IDF + length normalization) with top-K selection via nth_element + partial sort, achieving ~4.4K queries/sec on a 50K-document corpus.
  • Tradeoff: Chose tombstoning + periodic compaction for updates/deletes to keep query-time filtering stable and bound index bloat, trading compaction write amplification for predictable read/query behavior.
